# Система аутентификации и авторизации Django REST Framework

## Описание

Этот проект представляет собой полноценную систему аутентификации и авторизации на базе Django и Django REST Framework. Система реализует модель RBAC (Role-Based Access Control) с поддержкой ролей, прав доступа, JWT-токенов и мягкого удаления пользователей.

## Функциональные возможности

- **Аутентификация**: Регистрация, вход и выход пользователей с использованием JWT-токенов
- **Авторизация**: Гибкая система контроля доступа на основе ролей и прав
- **Управление профилем**: Просмотр и обновление данных профиля, мягкое удаление аккаунта
- **Административная панель**: Полнофункциональная админка для управления пользователями, ролями и правами
- **Мок-данные**: Заглушки для демонстрации работы системы с бизнес-объектами (статьи и комментарии)

## Архитектура

### Модели данных

Система основана на следующих моделях:

1. **User (Пользователь)**: Кастомная модель пользователя с поддержкой мягкого удаления
2. **Role (Роль)**: Определяет тип пользователя (admin, user, manager и др.)
3. **Permission (Право)**: Определяет конкретное действие над ресурсом (create, read, update, delete)
4. **RolePermission (Связь Роль-Право)**: Назначает права ролям
5. **UserRole (Связь Пользователь-Роль)**: Назначает роли пользователям

### Бизнес-логика

- **Аутентификация**: Реализована через DRF SimpleJWT с access и refresh токенами
- **Авторизация**: Проверка прав доступа через систему ролей и разрешений
- **Мягкое удаление**: Пользователи помечаются как удаленные, но их данные сохраняются
- **Административная панель**: Расширенная админка с фильтрами, поиском и управлением связями

## API Эндпоинты

### Аутентификация

- `POST /api/auth/register/` - Регистрация нового пользователя
- `POST /api/auth/login/` - Вход пользователя
- `POST /api/auth/logout/` - Выход пользователя

### Управление профилем

- `GET /api/users/profile/` - Получение данных профиля (автоматически)
- `PUT/PATCH /api/users/profile/` - Обновление профиля
- `DELETE /api/users/profile/delete/` - Мягкое удаление профиля

### Мок-данные (демонстрация)

- `GET /api/articles/` - Список статей (доступно всем авторизованным)
- `POST /api/articles/create/` - Создание статьи (требует роли admin или manager)
- `GET /api/comments/` - Список комментариев (доступно всем авторизованным)
- `POST /api/comments/create/` - Создание комментария (доступно всем авторизованным, проверка существования статьи)

## Зависимости

- Django 6.0.3
- Django REST Framework 3.16.1
- Django REST Framework SimpleJWT 5.5.1
- drf-yasg (Swagger UI)
- PostgreSQL (через psycopg2-binary)

## Установка и запуск

1. Клонируйте репозиторий
2. Установите зависимости: `poetry install`
3. Активируйте виртуальное окружение: `eval $(poetry env activate)`
4. Настройте базу данных в `config/settings.py`
5. Выполните миграции: `poetry run python manage.py migrate`
6. Создайте суперпользователя: `poetry run python manage.py createsuperuser`
7. Запустите сервер: `poetry run python manage.py runserver` или (в проект добавлено простейшее логирование): `poetry run python manage.py runserver 0.0.0.0:8000 2>&1 | tee logs.txt`

Основные настройки находятся в `config/settings.py`:

- **REST Framework**: Аутентификация через JWT, права доступа
- **SWAGGER**: Настройки документации API

## Тестовые данные

Система включает тестовые данные:

- **Роли**: admin, user, manager
- **Права**: create, read, update, delete для ресурсов Article и Comment
- **Пользователи**: admin@example.com, user1@example.com, manager1@example.com

### Работа с мок-данными

Мок-данные реализованы в файле `users/mock_views.py` и используются для демонстрации работы системы авторизации. Основные особенности:

1. **Создание статьи**: 
   - Доступно только пользователям с ролью admin или manager
   - Проверка прав осуществляется через кастомное разрешение `IsAdminOrManager`
   - Генерируется уникальный ID для каждой новой статьи
   - Новая статья добавляется в начало списка

2. **Создание комментария**:
   - Доступно всем авторизованным пользователям
   - Проверяется существование статьи по `article_id` перед созданием комментария
   - При попытке создать комментарий к несуществующей статье возвращается ошибка 404

3. **Просмотр данных**:
   - Список статей и комментариев доступен всем авторизованным пользователям
   - Данные возвращаются в формате JSON

Для демонстрации функционала можно использовать тестовые учетные записи через Swagger UI, проверяя различия в доступе в зависимости от роли пользователя.

## Особенности реализации

- **Кастомная модель пользователя**: Наследуется от AbstractBaseUser с email в качестве USERNAME_FIELD
- **Менеджер пользователей**: UserManager с методами для мягкого удаления
- **Мягкое удаление**: Пользователи не удаляются физически, а помечаются флагом is_deleted
- **Гибкая авторизация**: Права назначаются на уровне ресурсов и действий
- **Мок-данные**: Реализованы как заглушки для демонстрации работы системы

## Безопасность

- Пароли хешируются с использованием Django password hashing
- JWT-токены имеют ограниченное время жизни
- Все запросы к API требуют аутентификации (кроме регистрации и входа)
- Поддержка refresh токенов и их revocation

## Расширение функциональности

Систему можно расширить следующим образом:

1. Добавление новых ролей и прав через админку
2. Создание новых бизнес-объектов и соответствующих прав
3. Реализация email-верификации при регистрации
4. Добавление двухфакторной аутентификации
5. Интеграция с внешними провайдерами аутентификации (OAuth2, OpenID Connect)
